Word programming model

Microsoft Word is fully programmable. Some examples of this programmability were shown in the last article. Practically every feature that is provided by Word through its own menus and dialogs is available as a programmable command or function.

In order to simplify the programming process, a hierarchical object model has been evolved for Word. Following the object model is simple and intuitive. Once you understand the object model, you can actually predict how a particular task can be achieved using a specific command.

Incidentally, leaving aside the syntactical differences, the discussion conceptually applies to Excel, PowerPoint and MS Project also. Their object models are different and some of these will be covered later. However the base concept of utilising the capabilities of another application using VBA remain the same.

Using IntelliSense Help

Word VBA windows, like Visual Basic, provide full IntelliSense based help. Traditionally developers had to spend a lot of time and energy in memorising the exact spellings, syntax, data-types and arguments of the various commands and functions. IntelliSense eliminates the need to memorise all this.

The Word object model

Internally, Word is a very complex program. Therefore its object model is also complex. Here is a simplified version of the object model. For a detailed and complete discussion on the object model and its properties/methods, you can refer to the VBA help file which is included with Office 97. The filename is VBAWRD8.HLP and is available in the OFFICE directory of the Office CD. This diagram alongside illustrates some of the most important top-level objects. The topmost object is called the Application object and is the representation of the Word program itself. Under this object there are multiple objects grouped together based upon their functionality. Some of these objects themselves contain multiple objects and therefore are called collections.

The most important collection here is the documents collection. This collection contains references to all open documents. Each member of the Documents object is a single document using this high-level object model.

Consider the collection called Dialogs. This collection represents all user dialogs which can be invoked through Word. For example if you have an object reference to Word as objwrd1, you can programmatically invoke the Format - Style dialog using the command:

objwrd1.dialogs.item(wdDialogFormatStyle).display
Similarly the options property is a representation of parameters available on the Tools - Options dialog. Therefore if you wanted to, for example, enable the background printing option, you could use the command:

objwrd1.options.PrintBackground = True
Each document represents a single word-processor file. As a document contains words, characters, fonts, paragraphs, styles, formats and so on, the document itself has its own object model. This object model is very important for using Word as a report writer because you will need to directly manipulate a raw document object programmatically.

Now we have understood the base object model. Let us concentrate on common tasks that a programmer will need to perform.

Word features relevant to developers
Let us list some important usage scenarios where you as a developer would like to exploit some features of Word for enhancing your applications. The basic use of Word is to generate reports that are difficult to manage using standard report writing tools like built-in report writers of VFP, PB, D2K, etc

Form letters

Form letters are reports that are of a 'fill-in-the-blanks' type. These type of reports are very commonly used for statutory statements.

Programmatic mail-merge

If your application requires mail-merge features based upon the data stored under your system, it is easier to directly utilise Word's mail-merge features programmatically. Otherwise, the user has to understand the intricacies of mail-merge, based upon an ODBC data source. This is a complex and time-consuming procedure. If you know the content of the letter, it is very simple to use and generate mail-merge features programmatically.

Complex tables which require word wrap

The table handling features of Word are highly sophisticated. This makes Word an ideal solution for managing reports that require complex positioning of tables and text.

Reports where page properties are not fixed

Most report writers require you to set the page size at design time. If you change the style at runtime, the fields may not fit properly. Using Word for reporting allows you to change the page properties at runtime with ease.

Editable reports

Most reports, when generated can be viewed on screen or printer. If the user wants a WYSIWYG copy of it for editing, further reference, or e-mail, it is difficult to achieve. Some report writers like Crystal Reports allow you to save the generated report to a special file which is understood only by that reporting engine. However, providing editable reports is not possible.

Using Word/Excel for reporting solves this problem. You can not only give an editable version of the report, the users can save it themselves and manipulate the content using the powerful features of Word/Excel without you having to provide any additional functionality. For example, if you have generated a report in Excel, the user can perform a pivot table, draw a graph, perform trend analysis, etc, using the base features of Excel.

Saving a report as an HTML file

Most reporting engines do not support direct conversion to HTML. If your report is a Word document, you can simply we the Save as HTML functionality of Word. Word has a built-in converter for converting a DOC file to an HTML file.

Extended search and viewing features

Word incorporates a special feature called a Document Map. This shows a tree-like view of the various headings for easy navigation to any part of the report. The search dialog of Word is very powerful and can also search for specific formatting. Thus you can provide for features like 'jump to next subtotal' or 'next item with sales margin below 10%.' This provides for fast and easy searching capabilities, especially useful in a long report.

Conditional formatting

Sometimes, we may want a different set of formatting options to be applied to a row in the same column based upon some conditions. For example, you may want to highlight names of salespersons that have not met targets using a red, bold colour. Persons who have exceeded targets could be shown in blue. This is very easy to achieve in Word using Styles.

User-defined formatting changes to reports

Report writers are very rigid when it comes to providing runtime flexibility of formatting. Word reports can be implemented based upon Word templates. These templates are Word documents and can be edited by end-users simply using Word. Therefore changes like font or page size can be easily implemented at the end-user level without you having to change the base code.

Reports with custom features

Word documents are based upon templates. If you do not specify a template while creating a document, the documents are based upon the NORMAL.DOT template. A template can contain auto-text entries, macros, special commands and toolbars coded by you, and so on. Thus apart from using Word to render the report, you could base this report on a custom template that provides additional functionality.

For example, you could create a template that is used to generate the daily production report. You could provide a 'summarise' button on a custom toolbar. This would search through the contents of the document marked by a particular formatting style, and generate a small summary of the production data.

Word as a reports binder

Many of us require periodic reporting functionality. Every month we may need to submit a thick docket to the end-users containing a collection of multiple reports. As these reports are generated independently, a separate table of contents needs to be generated manually. This process can be automated using Word as a report writer. You can use the 'master document' functionality of Word to generate a base document which contains each individual report as an embedded sub-document. A combined table of contents can be created programmatically at the master document level.

This list of ideas is far from complete. However, it highlights the usability and power of using Word (and other Office products) as a very effective programming tool. In further articles I will provide the actual code required to implement some of the tasks listed above.
